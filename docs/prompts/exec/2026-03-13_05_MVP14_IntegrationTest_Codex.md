# 2026-03-13_05_MVP14_IntegrationTest_Codex.md

## 目的
COBOL → C# 変換プロジェクトにおいて **MVP14: I/O通し処理の統合テスト** を実装する。  
このプロンプトは **MVP14 実装本体** ではなく、**MVP12（READ）+ MVP13（WRITE）を結合した通し検証** を成立させるためのものです。

このMVPのゴールは次の通りです。

- `READ` → 処理 → `WRITE` の一連処理を自動テストで保証する
- 入力 fixture から期待出力 fixture までを再現可能に検証する
- EOF / 空ファイル / 1件 / 複数件 を通しで確認する
- 既存の MVP12 / MVP13 の責務境界を壊さずに統合できる
- 今後の I/O拡張（加工処理、集計処理、条件分岐付きI/O）の回帰基盤を作る

---

# 前提

このプロンプトを実行する前に、以下が最低限入っていること。

- MVP12: `READ ... AT END` の最小実装
- MVP13: `WRITE` の最小実装
- sequential input runtime
- sequential output runtime
- 最低限の I/O サンプルとテスト基盤

もし未実装・不安定な箇所があれば、まずそちらを補ってください。  
このプロンプトでは **統合テストと最小通しサンプル** に集中してください。

# 今回のスコープ

## 対応対象
- `OPEN INPUT`
- `OPEN OUTPUT`
- `READ ... AT END`
- `WRITE`
- `CLOSE`
- EOFまで繰り返す通し処理
- 入力レコードをそのまま出力する最小パターン
- 行数カウントのような最小処理を伴うパターン

## 非対応
今回は以下は実装対象外です。

- `READ INTO`
- `WRITE FROM`
- indexed file
- sort / merge
- 複雑な record layout
- 可変長レコード
- printer control
- file status 完全対応
- 例外系の高度なリカバリ

未対応構文は既存の未対応通知機構に従って、必ず観測可能にしてください。

# 実装方針

## 方針1: 統合テストを主成果物にする
今回は新機能を大きく足すのではなく、**MVP12 と MVP13 を組み合わせて成立することを証明する** のが主目的です。  
したがって、主な成果物は次です。

- 最小 COBOL サンプル
- 入力 fixture
- 期待出力 fixture
- 変換＋実行＋比較の統合テスト

## 方針2: まずは「そのまま通す」
最小の通しケースは、**入力レコードをそのまま出力** することです。  
これにより、I/O基盤が壊れていないことを最も明快に確認できます。

## 方針3: 次に「行数カウント」
余力があれば、または既存サンプルが作りやすければ、**行数カウント** のパターンも追加してください。  
これは EOFの終端制御が正しいことを確認しやすいです。

## 方針4: 最小差分
runtime や codegen に不足がある場合だけ最小補修を行ってください。  
統合テスト追加の名目で大規模な設計変更はしないでください。

## Task 2. 最小通しCOBOLサンプル追加
最低限、次の2系統を用意してください。

### ケースA: 入力をそのまま出力
意図:
- 1行ずつ `READ`
- EOF まで繰り返す
- 読めた行をそのまま `WRITE`

概念例:

```cobol
OPEN INPUT INFILE
OPEN OUTPUT OUTFILE

PERFORM UNTIL EOF-FLG = 'Y'
    READ INFILE
        AT END
            MOVE 'Y' TO EOF-FLG
        NOT AT END
            WRITE OUTREC
    END-READ
END-PERFORM

CLOSE INFILE
CLOSE OUTFILE
```

※ 実際には OUTREC へのMOVEが必要な既存仕様なら、それに合わせてください。  
重要なのは「入力→出力が壊れずに通ること」です。

### ケースB: 行数カウント
意図:
- EOFまで読み込む
- 読めた件数を加算する
- 最後に件数を出力または検証可能な形で観測する

## Task 3. fixture 追加
最低限、次の入力 fixture を追加してください。

### Fixture 1: 空ファイル
期待:
- 出力空
- EOFで正常終了

### Fixture 2: 1行
入力例:
```text
A
```

期待:
```text
A
```

### Fixture 3: 複数行
入力例:
```text
A
B
C
```

期待:
```text
A
B
C
```

可能なら以下も追加してください。

### Fixture 4: 空白含み
```text
A
HELLO WORLD
Z
```

改行・空白保持が既存仕様上どう扱われるか確認できるようにすること。

---

## Task 4. 統合テスト追加
既存のテスト基盤に合わせて、以下の統合テストを追加してください。

### 必須
1. 空ファイルで正常終了する
2. 1行入力がそのまま出力される
3. 複数行入力がそのまま出力される
4. EOFで無限ループしない
5. 既存MVP12 / MVP13 の単体テストを壊していない

### 推奨
6. 行数カウントが一致する
7. 変換後 C# のコードスナップショット比較
8. 標準出力または出力ファイル内容の golden 比較
9. Windows 改行 / Linux 改行差異に過度に依存しない比較

---

## Task 5. 必要最小限の補修
統合の過程で不足が見つかった場合のみ、次を最小限補修してください。

- input current record の受け渡し
- output record の設定
- OPEN / CLOSE の順序
- file object のスコープ
- EOF時の current record の扱い

ただし、今回の主目的は **統合成立の証明** です。  
設計を広げすぎないでください。
