# MVP14 Codex Prompt
Date: 2026-03-13
Target: COBOL → C# Migration MVP
Topic: Indexed File START

---

## 目的

MVP14として、COBOLの **START 文による Indexed file の位置決め** を最小構成で検証する。

このMVPで必須とする対象:

- `ORGANIZATION IS INDEXED`
- `RECORD KEY`
- `START ... KEY >= ...`
- `READ ... NEXT`

外部で Indexed ファイルを事前作成しなくてもよいように、
**プログラム内で最小の Indexed ファイルを作成してから START を実行する**。

2. `samples/data/mvp14/expected1.txt`
3. `samples/data/mvp14/expected2.txt`

## 実行時ファイルパスの扱い

Indexed ファイルのパス解決は次の優先順とする。

1. 環境変数 `MVP14_INDEX` が存在すればその値を使う
2. なければ固定名 `mvp14-index.dat` を使う

テスト側では、実行前に対象ファイルが存在する場合は削除してから実行すること。

テストケース切り替え:

- 環境変数 `MVP14_CASE` を使用する
- 値は `1` または `2`

## COBOLファイル定義

```cobol
SELECT IDX-FILE ASSIGN TO DYNAMIC WS-IDX-PATH
    ORGANIZATION IS INDEXED
    ACCESS MODE  IS DYNAMIC
    RECORD KEY   IS IDX-KEY
    FILE STATUS  IS WS-FS.

FD IDX-FILE.

01 IDX-REC.
   05 IDX-KEY  PIC 9(4).
   05 IDX-TEXT PIC X(10).
```

---

## WORKING-STORAGE

```cobol
01 WS-IDX-PATH PIC X(260).
01 WS-FS       PIC XX.
01 WS-CASE     PIC 9 VALUE 1.
01 WS-STARTKEY PIC 9(4).
01 WS-EOF      PIC X VALUE "N".
   88 EOF VALUE "Y".
```

## プログラム処理フロー

### Step 1 — Indexedファイルを作成する

```cobol
OPEN OUTPUT IDX-FILE
WRITE KEY=0001 TEXT="AAA"
WRITE KEY=0003 TEXT="CCC"
WRITE KEY=0005 TEXT="EEE"
CLOSE IDX-FILE
```

実際の実装では、レコード項目に `MOVE` してから `WRITE IDX-REC` すること。

---

### Step 2 — START 位置決め

```cobol
OPEN I-O IDX-FILE

IF WS-CASE = 1
   MOVE 3 TO WS-STARTKEY
ELSE
   MOVE 2 TO WS-STARTKEY
END-IF
```

続いて START を行う:

```cobol
START IDX-FILE KEY >= WS-STARTKEY
```

---

### Step 3 — READ NEXT で順次取得

ループで次を実行する:

```cobol
READ IDX-FILE NEXT
    AT END SET EOF TO TRUE
    NOT AT END
        DISPLAY "KEY=" IDX-KEY "|TEXT=" IDX-TEXT
END-READ
```

EOF になるまで繰り返すこと。

---

### Step 4 — 終了処理

```cobol
DISPLAY "DONE"
CLOSE IDX-FILE
GOBACK
```

- `START` を必ず使うこと
- `READ NEXT` を必ず続けて使うこと
- `ORGANIZATION IS INDEXED` を使うこと
- 余計な構文を入れないこと

## Codex への指示文（そのまま使用可）

```text
目的:
COBOL→C#変換MVP14として、START（キー指定でのポジショニング）を最小検証する。
対象は「ORGANIZATION IS INDEXED」「RECORD KEY」「START ... KEY >= ...」「READ ... NEXT」を必ず含むこと。
外部依存を避けるため、プログラム内で最小のIndexedファイルを作成してからSTARTを行う。
※本MVPの主目的はSTARTなので、派生（INVALID KEYなどの網羅）は最小に留める。

成果物:
1) samples/cobol/Mvp14Program.cbl を新規作成
2) samples/data/mvp14/expected1.txt を新規作成（複数行）
3) samples/data/mvp14/expected2.txt を新規作成（複数行）
4) apps/backend/tests に CobolMvpRuntimeMvp14Tests.cs を追加
   - ケース1/2をそれぞれ実行し、標準出力（複数行）を expected と完全一致比較
   - 既存のMvp03〜Mvp13のテスト構造・実行方法・改行コードに合わせる
5) diff（patch）形式で出力すること

重要: 実行時のファイルパス（テストで確実に成立させる）
- Indexedファイルのパスは次の優先順:
  1) 環境変数 MVP14_INDEX が存在すれば、そのパスを使用
  2) なければ固定名 "mvp14-index.dat"（作業ディレクトリ直下）
- テスト側は、実行前に "mvp14-index.dat" が存在するなら削除してから実行する。
- 実行は「ケース1」「ケース2」を切り替えられるようにする（環境変数 MVP14_CASE=1/2 など）。
  もしくは、テスト側で引数を渡せる仕組みが既存MVPにあるならそれに合わせる。

COBOL仕様（必須）:
- ファイル定義（Indexed）:
  SELECT IDX-FILE ASSIGN TO DYNAMIC WS-IDX-PATH
      ORGANIZATION IS INDEXED
      ACCESS MODE  IS DYNAMIC
      RECORD KEY   IS IDX-KEY
      FILE STATUS  IS WS-FS.

  FD IDX-FILE.
  01 IDX-REC.
     05 IDX-KEY  PIC 9(4).
     05 IDX-TEXT PIC X(10).

- WS:
  01 WS-IDX-PATH PIC X(260).
  01 WS-FS       PIC XX.
  01 WS-CASE     PIC 9 VALUE 1.
  01 WS-STARTKEY PIC 9(4) VALUE 0.
  01 WS-EOF      PIC X VALUE "N".
     88 EOF VALUE "Y".
     88 NOT-EOF VALUE "N".

処理全体（必須フロー）:
A) Indexedファイルを作る（プログラム内で完結させる）
  1) WS-IDX-PATH を決定（MVP14_INDEX or "mvp14-index.dat"）
  2) OPEN OUTPUT IDX-FILE
  3) 次の3レコードをWRITE（キー順に書く）:
      KEY=0001 TEXT="AAA"
      KEY=0003 TEXT="CCC"
      KEY=0005 TEXT="EEE"
     ※文字列は固定でよい。必要ならMOVEで設定。
  4) CLOSE IDX-FILE

B) STARTしてREAD NEXTで取り出す（ここがMVPの核）
  5) OPEN I-O IDX-FILE  （または INPUT でもよいが、START+READ NEXTが可能な形にする）
  6) ケースに応じて WS-STARTKEY を設定:
      - ケース1: WS-STARTKEY = 0003
      - ケース2: WS-STARTKEY = 0002
  7) START IDX-FILE KEY >= WS-STARTKEY
       INVALID KEY の場合は "START=INVALID" を表示して終了でもよいが、
       今回は上のデータなら通常はINVALIDにならない想定（最小）
  8) START後、READ IDX-FILE NEXT を繰り返し、読めたレコードをDISPLAYする:
      DISPLAY "KEY=" IDX-KEY "|TEXT=" IDX-TEXT
     EOFになったらループ終了。
  9) CLOSE IDX-FILE
 10) 最後に DISPLAY "DONE"

出力仕様（完全一致比較用）:
- ケース1（STARTKEY=0003）では、0003と0005が出る:
  KEY=0003|TEXT=CCC
  KEY=0005|TEXT=EEE
  DONE

- ケース2（STARTKEY=0002）では、最初の >= が0003なので同じく0003と0005が出る:
  KEY=0003|TEXT=CCC
  KEY=0005|TEXT=EEE
  DONE

テストデータ（expected固定）:
- expected1.txt（ケース1）:
  KEY=0003|TEXT=CCC
  KEY=0005|TEXT=EEE
  DONE

- expected2.txt（ケース2）:
  KEY=0003|TEXT=CCC
  KEY=0005|TEXT=EEE
  DONE

制約:
- STARTが主目的なので、複雑な例外網羅はしない（最小）
- AFTER ADVANCING 等は使わない
- 既存のリポジトリ規約（命名、配置）を厳守する
- テストは標準出力を expected と完全一致比較すること
```
